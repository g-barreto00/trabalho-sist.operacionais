Módulo 02

Processos

O modelo de processo é a abstração central dos sistemas operacionais modernos.

Um processo é uma instância de um programa em execução. Enquanto o programa é um arquivo estático em disco — apenas um algoritmo expresso em código —, o processo é dinâmico: possui estado próprio, contador de programa e recursos alocados pelo SO.

A distinção é sutil, mas crucial. Tanenbaum ilustra bem: imagine um cientista seguindo uma receita de bolo. A receita é o programa, o cientista é o processador e o ato de cozinhar é o processo. Se o filho chega chorando com uma picada de abelha, o cientista salva seu estado (onde estava na receita) e passa a seguir o livro de primeiros socorros — um processo mais prioritário. Quando termina, retoma o bolo do ponto onde parou.

Vale notar: se o mesmo programa for executado duas vezes simultaneamente, isso resulta em dois processos distintos, mesmo que compartilhem o mesmo código em memória.

Cada processo contém:

  • Código (text) — instruções do programa
  • Dados (data/heap) — variáveis globais e alocação dinâmica
  • Pilha (stack) — variáveis locais e retorno de funções
  • PCB — Process Control Block com metadados do SO
Multiprogramação

Em sistemas modernos, a CPU alterna rapidamente entre vários processos — cada um por dezenas ou centenas de milissegundos — criando a ilusão de paralelismo. Esse mecanismo é chamado de multiprogramação. Na prática, em qualquer instante apenas um processo usa a CPU, mas ao longo do tempo todos avançam.

Um processo transita entre três estados fundamentais ao longo de sua vida. A distinção mais importante é entre os dois primeiros: nos estados Em Execução e Pronto, o processo está disposto a executar — a diferença é apenas a disponibilidade da CPU. O estado Bloqueado é fundamentalmente diferente: o processo não pode executar mesmo que a CPU esteja ociosa.

Em Execução (Running) — usando a CPU naquele instante.
Pronto (Ready) — apto a executar, aguardando a CPU ser alocada pelo escalonador.
Bloqueado (Waiting) — incapaz de executar até que um evento externo aconteça, como a chegada de uma entrada.
Diagrama de Transições
1
Running → Blocked — o processo não pode continuar; está esperando por uma entrada ou evento. Causada pelo próprio processo.
2
Running → Ready — o escalonador decide que o processo usou CPU por tempo suficiente. Causada pelo escalonador, sem que o processo saiba.
3
Ready → Running — o escalonador seleciona este processo para receber a CPU. Também causada pelo escalonador.
4
Blocked → Ready — o evento esperado ocorreu. O processo volta à fila de prontos e aguarda o escalonador selecioná-lo.

Nota: repare que não existe transição direta de Bloqueado para Em Execução — o processo sempre passa pelo estado Pronto antes de receber a CPU novamente.

Quando o escalonador decide trocar o processo em execução, o SO realiza uma troca de contexto: salva o estado do processo atual e restaura o estado do próximo. Esse mecanismo é o que permite a ilusão de que vários processos executam simultaneamente.

Registradores gerais — valores que o processo estava usando na CPU.
Contador de programa — qual instrução executar quando o processo voltar.
Ponteiro de pilha — posição atual da pilha de execução.
Palavra de estado (PSW) — flags e modo de execução do processo.
Como ocorre na prática
1
O hardware salva o contador de programa na pilha ao detectar a interrupção.
2
Uma rotina em linguagem de montagem salva os registradores na tabela de processos.
3
O escalonador decide qual processo executará a seguir.
4
Os registradores e o mapa de memória do novo processo são carregados — e ele retoma de onde parou.

Custo: durante a troca, a CPU não executa trabalho útil. Além disso, o cache e a TLB podem ser invalidados, forçando recarregamentos da memória — por isso trocas de contexto em excesso prejudicam o desempenho.

Hierarquia de Processos
fork()
Cria uma cópia exata do processo pai — memória, registradores, arquivos abertos e tudo mais. Após o fork, pai e filho seguem caminhos independentes. O retorno é 0 no filho e o PID do filho no pai, permitindo que cada um saiba seu papel.
exec()
Substitui a imagem de memória do processo atual por um novo programa. Usado logo após fork() quando o filho precisa executar algo diferente do pai — é assim que o shell executa comandos.
init
Processo raiz de todo o sistema UNIX. Ao inicializar, cria processos para cada terminal e é ancestral de todos os processos em execução — formando uma única árvore.
Módulo 03

Threads

Threads permitem que um único processo realize múltiplas tarefas concorrentemente.

bolt

Vantagens das Threads

Por que usar threads em vez de múltiplos processos? Os livros clássicos destacam quatro benefícios fundamentais:

Responsividade — o programa continua a responder ao usuário mesmo que uma thread esteja bloqueada ou executando uma operação demorada.
Partilha de Recursos — threads compartilham a memória e os recursos do processo por padrão, facilitando a comunicação entre elas.
Economia — criação e gestão de threads consomem significativamente menos memória e tempo de CPU do que criar processos inteiros.
Escalabilidade — em arquiteturas multicore, threads podem correr em paralelo em diferentes núcleos, aumentando drasticamente o desempenho.

Processo vs Thread

Processos têm espaço de endereçamento independente. Threads de um mesmo processo compartilham memória, o que torna a comunicação mais rápida — mas exige sincronização cuidadosa.

Nota — Troca de Contexto entre threads A troca de contexto entre threads do mesmo processo é muito mais leve do que entre processos diferentes. Como compartilham o mesmo espaço de endereçamento, o SO não precisa trocar as tabelas de páginas nem limpar a TLB (Translation Lookaside Buffer), poupando tempo valioso de processamento.
layers

Modelos de Thread

Tipos de Implementação
Threads de Utilizador — geridas por biblioteca, rápidas de criar, mas o bloqueio de uma paralisa todo o processo.
Threads de Kernel — geridas pelo SO, mais overhead na criação, mas permitem paralelismo real entre threads.
Modelos de Mapeamento
Many-to-One

Várias threads de usuário mapeadas para um único thread de kernel. Simples, mas sem paralelismo real.

One-to-One

Cada thread de usuário tem um thread de kernel. Mais paralelismo, mais overhead. Usado no Linux e Windows.

Many-to-Many

Vários threads de usuário mapeados para vários de kernel. Flexível, mas complexo de implementar.

sync_lock

Sincronização

O acesso concorrente a recursos compartilhados pode causar condições de corrida (race conditions). As soluções incluem:

Mutex — exclusão mútua; só uma thread acessa por vez.
Semáforo — contador que controla acesso a recursos limitados.
Monitor — abstração de alto nível com locks e variáveis de condição.
Módulo 04

Algoritmos de Escalonamento

Nota: estes trechos são versões didáticas, escritas com arrays simples (ao invés de classes/objetos) para focar na lógica do algoritmo. O simulador implementado neste projeto usa uma estrutura de dados diferente, mais adequada à aplicação real.

First In, First Out — o primeiro processo a chegar é o primeiro a ser executado. Simples de implementar, mas sofre do problema do efeito comboio: um processo longo bloqueia todos os outros.

Vantagens
  • ✓ Simples de implementar
  • ✓ Sem starvation
Desvantagens
  • ✗ Efeito comboio
  • ✗ Espera média alta
Python 
from collections import deque

fila = deque(["P1", "P2", "P3"])

while fila:
    proximo = fila.popleft()   # Primeiro que entrou, primeiro que sai
    print(proximo)

# Saída:
# P1
# P2
# P3

Shortest Job First — o processo com menor tempo de execução é selecionado primeiro. Minimiza o tempo médio de espera, sendo ótimo nesse critério. O problema é que o tempo de execução geralmente não é conhecido com antecedência.

Vantagens
  • ✓ Minimiza espera média
  • ✓ Ótimo (se burst conhecido)
Desvantagens
  • ✗ Burst geralmente desconhecido
  • ✗ Starvation de processos longos
Python 
prontos = [("P1", 5), ("P2", 4), ("P3", 2)]   # (processo, burst)

while prontos:
    prontos.sort(key=lambda p: p[1])   # ordena pelo menor burst disponível
    pid, burst = prontos.pop(0)        # escolhe o mais curto
    print(pid)

# Saída:
# P3   (burst 2)
# P2   (burst 4)
# P1   (burst 5)

Cada processo recebe uma prioridade numérica. O processo de maior prioridade sempre é executado. Pode ser estático (definido na criação) ou dinâmico (ajustado pelo SO). Risco de starvation — processos de baixa prioridade nunca executam.

A solução clássica para starvation é o envelhecimento (aging): a prioridade de um processo aumenta progressivamente enquanto ele aguarda.

Python 
prontos = [("P1", 2), ("P2", 1), ("P3", 3)]   # (processo, prioridade)

while prontos:
    prontos.sort(key=lambda p: p[1])   # menor número = maior prioridade
    pid, prio = prontos.pop(0)         # escolhe o de maior prioridade
    print(pid)

# Saída:
# P2   (prioridade 1)
# P1   (prioridade 2)
# P3   (prioridade 3)

Cada processo recebe um quantum de tempo fixo. Ao esgotar o quantum, o processo é preemptado e vai para o fim da fila, cedendo a CPU ao próximo. É o algoritmo base dos sistemas de tempo compartilhado.

A escolha do quantum é crítica: muito pequeno → muitas trocas de contexto; muito grande → degrada para FIFO.

Critério de escolha do Quantum

Regra geral: 80% dos bursts de CPU devem ser menores que o quantum. Tipicamente entre 10 ms e 100 ms.

Python 
from collections import deque

QUANTUM = 2
fila    = deque([("P1", 5), ("P2", 4), ("P3", 2)])  # (processo, burst)

while fila:
    pid, burst = fila.popleft()        # Retira o primeiro
    fatia       = min(QUANTUM, burst)  # Executa no máximo QUANTUM unidades
    burst      -= fatia

    if burst > 0:
        fila.append((pid, burst))      # Ainda tem burst → volta ao fim da fila
    else:
        print(f"{pid} concluído")

# Saída:
# P3 concluído
# P2 concluído
# P1 concluído

Earliest Deadline First — o processo com o deadline mais próximo sempre recebe a CPU. É um algoritmo ótimo para sistemas de tempo real: se existir uma escala viável (que respeite todos os deadlines), o EDF a encontrará.

A preempção ocorre sempre que um novo processo chega com deadline menor que o do processo em execução. Processos sem deadline são tratados como deadline infinito e ficam com a menor prioridade.

Vantagens
  • ✓ Ótimo para tempo real
  • ✓ Utilização de CPU até 100%
  • ✓ Minimiza deadlines perdidos
Desvantagens
  • ✗ Requer deadlines conhecidos
  • ✗ Overhead de reescalonamento
  • ✗ Falha catastrófica se sobrecarregado
Python 
prontos = [("P1", 8), ("P2", 12), ("P3", 5)]   # (processo, deadline)

while prontos:
    pid, dl = min(prontos, key=lambda p: p[1])  # escolhe o menor deadline
    prontos.remove((pid, dl))
    print(f"{pid}  deadline={dl}")

# Saída:
# P3  deadline=5
# P1  deadline=8
# P2  deadline=12
